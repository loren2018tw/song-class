## Context

目前教師端已具備小白板元件與學生名單，但白板僅在教師端本地呈現。學生端仍停留在「請專心學習」首頁，且未接收教師塗鴉事件。這次變更需要把教師白板狀態（首頁/白板）與繪製內容同步到所有已連線學生，並在新學生加入時提供可重現當前畫面的初始化資料。

## Goals / Non-Goals

**Goals:**

- 教師端新增「首頁」按鈕，能明確切回首頁模式並同步所有學生端。
- 教師端切換「小白版」時，已連線學生端同步切換為白板觀看模式。
- 教師塗鴉透過 WebRTC DataChannel 傳送增量事件，學生端可逐步重現畫布。
- 新加入學生在連線完成後可收到白板快照，重現教師先前已繪製內容，再持續接收增量更新。
- 保持前端元件可重用結構，避免在教師端與學生端重複實作白板重播邏輯。

**Non-Goals:**

- 不在這次變更中加入學生端反向畫筆或多人共筆。
- 不在這次變更中實作白板資料長期持久化（例如資料庫儲存）。
- 不更動教師端白板核心繪圖 API（畫筆顏色、粗細、背景切換）的既有互動語意。

## Decisions

1. 採用「模式控制 + 白板事件」雙通道語意（同一 DataChannel 事件型別分流）。
   模式控制事件負責首頁/白板切換，白板事件負責增量繪製與快照同步。這樣可避免 UI 模式與繪圖資料耦合，降低事件處理分支複雜度。

2. 增量同步以筆畫操作事件為主，並採批次封包傳送。
   已連線學生只接收增量事件可減少頻寬；為支援預設同時約 30 位學生，教師端需將高頻增量事件以批次封包送出，學生端再依序重播，兼顧效能與一致性。

3. 教師端維護白板同步狀態快取（最近快照 + 事件序號）。
   快取讓新加入者初始化不需回放所有歷史事件，並可用事件序號避免重複套用或順序錯亂。

4. 學生端白板採「唯讀重播模式」。
   學生端共用白板渲染元件，但停用輸入事件，只保留套用快照與增量事件能力，確保互動權限由教師端單向控制。

5. 模式切換優先級高於繪製事件，且切回首頁時保留白板快照。
   收到首頁模式時，學生端先切換顯示層，再清理白板渲染流程中的暫存增量佇列，避免返回首頁後仍短暫閃現新筆畫。
   教師端與學生端都保留最近白板快照，下一次切回白板時直接恢復原畫面，不需重新傳送整份白板資料。

## Risks / Trade-offs

- [快照體積過大造成加入延遲] → 先限制快照僅包含必要背景與筆畫資料，避免攜帶可推導欄位。
- [增量事件丟失導致學生畫面不一致] → 事件帶序號，若發現序號跳號則向教師端請求重送最新快照。
- [多名學生同時在線造成傳輸壓力] → 增量事件採批次封包，並以單次序列化結果分發給預設約 30 位在線學生。
- [首頁與白板切換期間的競態] → 在模式事件中附帶模式版本號，學生端僅套用最新版本，捨棄過期事件。

## Migration Plan

1. 擴充白板事件型別與 DataChannel 訊息路由，加入模式切換、增量事件、快照事件。
2. 在教師端功能按鈕列新增「首頁」按鈕，並把模式切換事件廣播給所有已連線學生。
3. 調整學生端主畫面，加入首頁/白板兩種顯示狀態與唯讀白板重播。
4. 在學生完成連線流程時，教師端主動推送一次快照與目前模式，再接續增量批次傳送。
5. 以手動驗證覆蓋：已連線同步切換、新加入重現歷史、首頁切回一致性。

## Open Questions

- 批次封包的時間窗與單批最大事件數門檻（例如 16-33ms）需在實作階段依實測延遲微調。
- 若學生端偵測到序號跳號，是否立即請求最新快照或先容忍短暫缺包需在實作階段確認。
