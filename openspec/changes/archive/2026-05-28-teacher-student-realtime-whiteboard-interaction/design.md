## Context

現況白板流程為教師端單一白板畫面，學生端僅接收教師白板同步，未提供學生作答白板與教師集中檢視。此次變更需要在不破壞既有首頁/白板模式切換的前提下，擴充為「教師白板 / 學生白板」雙分頁互動，並讓教師端可同時查看所有學生白板。主要限制包含：

- 現有同步架構以 WebRTC DataChannel 為主，需維持事件序一致性。
- 現有學生端白板在白板模式為唯讀，需要新增可編輯模式並支援自行切換分頁。
- 教師端需在有限視窗內展示多位學生白板，版面需隨連線數變化。

## Goals / Non-Goals

**Goals:**

- 建立教師端雙分頁白板流程，且教師分頁切換不強制推送學生端切頁。
- 建立學生端可自主切換雙分頁呈現：教師白板唯讀、學生白板可繪圖。
- 建立學生到教師的批次增量回傳通道，使教師可重播每位學生白板。
- 建立教師端學生白板總覽網格，依在線人數自動縮放，縮圖不顯示工具列但顯示學生暱稱。

**Non-Goals:**

- 不納入學生與學生之間互看白板或互相批註功能。
- 不納入白板資料持久化到資料庫或課後歷史回放。
- 不調整 Rust signaling 伺服器的角色模型（仍維持 teacher/student 角色）。

## Decisions

1. 以「課堂模式 + 白板分頁」雙層狀態建模

- 決策：保留既有 `home` / `whiteboard` 課堂模式，於 `whiteboard` 模式下新增 `teacher-board` / `student-board` 分頁狀態。
- 理由：可最小化既有首頁切換邏輯變動，並清楚區分課堂層切換與白板子模式。
- 替代方案：直接把模式展平成三種狀態（home、teacher-board、student-board）。此作法會增加舊流程遷移風險，故不採用。

2. 事件通道採「方向化事件」

- 決策：在 DataChannel 訊息型別加入來源與目標語意，將事件分為教師下行（模式、教師白板快照/增量）與學生上行（學生白板增量批次）。
- 理由：可沿用現有批次與序號機制，並避免教師與學生事件混流造成重播錯誤。
- 替代方案：為學生上行另開獨立 DataChannel。會增加協商複雜度與錯誤面，故不採用。

3. 教師端以「每學生一份快照 + 序號」維護狀態

- 決策：教師端為每個 `connection_id` 維護獨立學生白板快照、最後序號與暫存批次隊列。
- 理由：可精準重播與容錯（序號落差時可要求補快照或重置該學生）。
- 替代方案：僅維護畫布位圖不保留結構化快照。將降低除錯與同步恢復能力，故不採用。

4. 教師端學生白板總覽採 CSS Grid 自適應

- 決策：使用 CSS Grid + `repeat(auto-fit, minmax(...))` 呈現學生白板縮圖，白板元件以唯讀且隱藏工具/標題模式渲染。
- 理由：可隨學生人數自動換行與縮放，且不需手動定義大量斷點。
- 替代方案：固定 2x2、3x3 版型。對人數變動彈性不足，故不採用。

5. 學生端分頁顯示為「自主可切換元件」

- 決策：學生端顯示可互動 tabs，允許學生在白板模式下自行切換教師白板/學生白板分頁。
- 理由：符合課堂互動節奏，學生可自行切換參考教師示範與作答畫面。
- 替代方案：由教師端強制同步分頁。會降低學生操作彈性且增加切頁同步訊息成本，故不採用。

6. 教師端分頁切換採狀態保留而非重建

- 決策：教師端「教師白板 / 學生白板」切換時，兩分頁白板狀態皆保留於記憶體，不以重新掛載元件作為切換手段。
- 理由：教師實際操作會頻繁切頁，若每次重建元件會造成白板快照遺失風險與畫面閃爍。
- 替代方案：每次切頁時重建白板元件並回補快照。會增加同步與渲染抖動風險，故不採用。

7. 教師端學生白板操作僅支援全清除

- 決策：本次僅實作「清空全部學生白板」，不實作清空單一學生白板。
- 理由：單一學生白板清除通常會與遠端定位、單人編輯檢視等能力綁定，超出本次範圍。
- 替代方案：先行提供單一清除按鈕。可能導致後續遠端編輯能力語意衝突，故不採用。

8. 教師端分頁切換不強制同步學生分頁

- 決策：教師端切換「教師白板 / 學生白板」時不送出學生端分頁修正訊息。
- 理由：避免不必要的控制訊息，並尊重學生端自主切頁行為。
- 替代方案：每次教師切頁都廣播分頁同步。會覆寫學生端操作且增加通道噪音，故不採用。

## Risks / Trade-offs

- [風險] 學生端高頻回傳造成教師端重播壅塞 → 以固定批次大小與 flush 週期控制，必要時對單學生套用背壓與丟棄過舊中間點策略。
- [風險] 多學生同步時記憶體成長過快 → 對每位學生快照限制最大筆畫數，並提供教師清空全部學生白板操作。
- [風險] 分頁切換瞬間與事件批次交錯導致套用錯頁 → 事件加上分頁版本號，僅在匹配版本下套用。
- [風險] 教師頻繁切頁造成白板元件重建與資料遺失 → 切頁採狀態保留策略，避免白板元件反覆卸載重建。
- [取捨] 教師端同屏顯示全部白板會降低單張可視尺寸 → 優先完整總覽，詳細檢視留待後續變更加入放大檢視。

## Migration Plan

1. 先擴充前端型別與事件協定（向後相容欄位，未啟用前不影響舊流程）。
2. 實作教師端雙分頁 UI（不強制學生切頁），再接上學生端自主分頁渲染。
3. 啟用學生上行批次事件與教師端學生白板重播儲存。
4. 最後切換教師端學生白板總覽 UI，並進行多人連線壓力測試。
5. 若需回滾，可關閉學生上行事件處理並強制教師端固定在教師白板分頁。
