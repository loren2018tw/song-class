## Context

目前 `WhiteboardCanvas` 以單一 `<canvas>` 同時承擔背景與塗鴉，初始化時會填滿白色背景，橡皮擦則透過「背景色畫筆」達成視覺覆蓋。這個模型在無背景圖時可用，但在導入教師可切換背景圖片後，會造成下列問題：

1. 橡皮擦會破壞背景（或需要耦合背景色），不符合「只擦筆跡」預期。
2. 背景與塗鴉資料混在同一層，不利於重繪與同步。
3. 元件缺乏標準化的外部背景輸入契約，父元件難以維持通用整合。

本次變更橫跨 `src/components/WhiteboardCanvas.vue`、`src/views/TeacherView.vue` 與白板型別，屬於既有能力的行為級變更。

## Goals / Non-Goals

**Goals:**

- 將白板改為背景層與塗鴉層分離，塗鴉層維持透明。
- 橡皮擦改為真正擦除已繪製筆跡，不影響背景圖。
- 提供父元件傳入背景圖片資料並觸發背景層重繪的通用介面。
- 教師端「教師背景」下拉選單變更時，可即時反映在白板背景層。
- 保持既有畫布座標系（1280×720）與縮放行為不變。

**Non-Goals:**

- 不新增圖片上傳流程或背景素材管理後端。
- 不改動學生端觀看模式的整體佈局。
- 不在本次變更中重做白板同步傳輸協定（僅做必要欄位延伸）。

## Decisions

### 1) 採用雙 Canvas 疊層（background + drawing）

**Decision:** 在同一 stage 容器中放置兩個同尺寸 canvas：底層背景 canvas、上層塗鴉 canvas。所有筆畫只寫入上層。

**Rationale:**

- 可完全分離背景與筆畫渲染責任。
- 清除塗鴉、回放筆畫與切換背景互不干擾。
- 不影響既有座標換算與縮放策略。

**Alternatives considered:**

- 單 canvas 每次重繪先畫背景再畫筆畫：可行但橡皮擦語意仍易與背景耦合，且後續擴充圖層能力成本較高。

### 2) 橡皮擦使用 `globalCompositeOperation = "destination-out"`

**Decision:** 橡皮擦在塗鴉 canvas 以 `destination-out` 繪製，直接移除像素而非覆蓋顏色。

**Rationale:**

- 能達成真正擦除效果。
- 對透明塗鴉層語意正確，背景層天然保留。

**Alternatives considered:**

- 透過重算 stroke list（把碰撞到的筆畫裁切/刪除）：資料結構與命中測試複雜度過高，不適合作為本次第一版。

### 3) 背景輸入採「父元件資料驅動」契約

**Decision:** 在白板元件新增背景輸入屬性（例如 `backgroundImage`，可接受 data URL / URL / null），由父元件控制背景來源。元件內提供單一路徑的背景重繪函式，當輸入變更時重繪背景層。

**Rationale:**

- 元件保持通用，不耦合教師專用素材來源。
- 便於後續在 Web 教師端、Tauri 視窗端共用同元件。

**Alternatives considered:**

- 由元件自行管理背景清單與下拉：會讓元件承擔頁面層責任，降低可重用性。

### 4) 快照資料分離背景與筆畫

**Decision:** 既有塗鴉快照維持筆畫資料為主，背景資訊改為獨立欄位（可為可選），序列化與還原流程須明確區分「背景重繪」與「筆畫重放」。

**Rationale:**

- 避免橡皮擦/清除畫布誤操作到背景。
- 有利於遠端同步時差異化更新（僅背景變更或僅筆畫變更）。

**Alternatives considered:**

- 保持背景不入快照、僅本地 UI 狀態：會造成遠端還原不一致風險。

### 5) 教師背景下拉由頁面層驅動白板背景

**Decision:** `TeacherView` 保留「教師背景」下拉為唯一背景來源，選項變更後更新傳給白板元件的背景資料。

**Rationale:**

- 責任分離清楚：頁面決策、元件渲染。
- 與現有教師操作流程一致，改動範圍可控。

**Alternatives considered:**

- 透過全域 store 廣播背景：目前需求單純，導入全域狀態會增加維護成本。

## Risks / Trade-offs

- [Risk] 雙層 canvas 可能導致重繪次數增加 → Mitigation：背景僅在輸入變更時重繪，筆畫仍維持增量繪製與必要時完整重畫。
- [Risk] 外部圖片載入失敗（跨網域或 URL 無效） → Mitigation：背景載入錯誤時回退為透明/預設底色並記錄可觀測錯誤訊息。
- [Risk] 舊快照資料缺少背景欄位 → Mitigation：型別設計採向後相容（背景欄位 optional），缺省時以無背景處理。
- [Trade-off] `destination-out` 只作用於像素層，不保留結構化擦除歷史 → Mitigation：本次先滿足互動需求，未來若需高精度可再升級為向量級擦除模型。

## Migration Plan

1. 先在 `WhiteboardCanvas` 落地雙層 canvas 與背景輸入，保持既有父層 API 可運作。
2. 調整橡皮擦實作為 `destination-out`，並以手動測試驗證不影響背景層。
3. 擴充白板型別與快照還原流程，確保舊資料可讀。
4. 串接 `TeacherView` 背景下拉到白板元件新輸入。
5. 完成教師端情境驗證（切換背景、繪圖、擦除、清除、同步還原）。
6. 若部署後發生問題，可暫時回退為「固定背景 + 單層畫布」版本（以 Git 變更回滾）。

## Open Questions

- 背景資料是否需要納入既有即時同步事件，或由教師端單向控制即可？
- 教師背景選單的選項來源（內建資源、遠端 URL、或資料 URI）在本階段是否已有固定規格？
- 清除畫布是否應只清除塗鴉層（建議），或同時清空背景（目前規劃為不清背景）？
