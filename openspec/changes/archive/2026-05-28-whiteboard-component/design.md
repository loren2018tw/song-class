## Context

目前教師端瀏覽器頁面只有學生清單，尚未提供可直接在課堂中塗繪的主畫面。這次變更要在教師端頁面中加入小白板能力，並讓白板狀態可以被父元件接收、轉送到遠端或重新還原。

## Goals / Non-Goals

**Goals:**

- 在教師端頁面提供可切換的小白板功能區。
- 讓白板以固定 1280×720 的內部座標系繪製，並在任意容器中縮放顯示。
- 支援滑鼠與觸控的連續塗繪，以及顏色、粗細、橡皮擦與清除功能。
- 將塗繪內容整理成可序列化 JSON，並能透過事件回傳父元件。
- 讓父元件可把同一份 JSON 送到其他白板實例以重現內容。

**Non-Goals:**

- 不在這次變更中實作課堂白板的多人即時同步協議。
- 不處理塗繪內容的伺服器持久化。
- 不加入複雜筆刷、圖形工具、文字工具或圖層系統。

## Decisions

1. 採用原生 `canvas` 作為繪圖核心，而不是 SVG 或 DOM 路徑。
   原因是白板主要需求是自由手繪，`canvas` 對連續線段、橡皮擦與高頻輸入的處理較直接，也比較容易維持固定 1280×720 座標系。
   其他方案如 SVG 雖然較容易序列化，但在大量筆畫時會比較重，且即時重繪與橡皮擦處理不如 canvas 直觀。

2. 內部資料以完整快照為主，輸出可重建的 JSON。
   白板資料會包含畫布尺寸、目前工具狀態與筆畫陣列。每一筆畫保留顏色、粗細、點位與工具類型，讓父元件可以把快照直接傳給另一個白板實例重畫。
   相較於只輸出增量指令，完整快照比較容易除錯，也更符合「可重現塗鴉內容」的需求。

3. 使用 pointer 事件統一處理滑鼠、觸控與筆輸入。
   透過 `pointerdown`、`pointermove`、`pointerup` 與 `pointercancel` 可以同時支援桌機與行動裝置，並且減少兩套輸入邏輯重複。
   其他做法如分開監聽 mouse/touch 事件會增加狀態同步成本。

4. 白板元件提供受控輸入與事件輸出。
   元件會接受一份可選的 snapshot prop，並在本地狀態變更後透過事件回傳更新後的 snapshot。這樣父元件既可以本地保存，也可以把相同資料送到遠端白板實例。
   事件以「資料回傳」為主，不讓元件直接決定同步目的地，降低耦合。

5. 顯示縮放交由外層容器控制，繪圖座標仍維持內部固定尺寸。
   UI 會以 16:9 容器呈現，畫布實際繪製仍落在 1280×720 座標系。滑鼠與觸控位置會依容器實際大小換算回內部座標，確保重播時結果一致。

## Risks / Trade-offs

- [座標換算誤差] → 使用單一座標轉換函式，並以畫布實際 bounding box 計算比例，避免在縮放時累積偏差。
- [大量筆畫造成重繪成本增加] → 先以完整快照與整體重繪實作，若後續筆畫量變大，再考慮離屏 canvas 或分段重繪。
- [橡皮擦語意不夠直觀] → 先將橡皮擦定義為白色筆畫，且粗細自動加 10 點，讓行為簡單一致。
- [父元件接收資料後如何同步到遠端] → 這次只保證輸出資料結構可重現，遠端傳輸流程留給後續整合。

## Migration Plan

1. 先新增白板元件與相關型別，不影響既有學生清單與連線流程。
2. 再把教師端瀏覽器頁面改成上方功能列、左下學生清單、右下白板主畫面。
3. 最後把白板快照事件接到父元件，保留後續同步或儲存的擴充點。
4. 若需回滾，只要移除白板元件掛載與功能切換即可恢復現有教師頁面。

## Open Questions

- 白板快照是否需要立即進入 WebSocket 同步，還是先只在父元件內流轉即可。
- 白板工具列是否需要預設顏色或筆粗細記憶到下一次開啟。
- 是否要在不同白板實例之間保留同一份畫布內容作為課堂狀態。
