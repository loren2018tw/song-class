## Context

目前系統已有教師/學生連線與即時同步基礎能力，主要互動場景聚焦在白板。此次需在既有課堂會話中加入「快問快答」流程，並滿足以下限制：

- 教師可在題目與選項文字留白的前提下發題，代表資料欄位可為空字串。
- 作答結束後先顯示彙總統計，個別學生答案需由教師主動切換才顯示。
- 教師可選擇是否設定正確答案；未設定時學生端不得顯示對錯判定。
- 教師尚未結束作答前，學生可隨時更換答案，統計需以每位學生最後一次作答計算。
- 功能需以瀏覽器教師端與瀏覽器學生端為唯一操作介面，主程式僅作為 server 控制台，並透過既有即時通道同步。

## Goals / Non-Goals

**Goals:**

- 提供完整快問快答生命週期：出題、作答中、結束、結果呈現。
- 定義可序列化的事件與資料結構，確保教師端與學生端狀態一致。
- 讓教師端可在同一頁面或同一模組完成統計總覽與明細切換。
- 在有設定正確答案時，提供學生端清楚且即時的答題回饋。

**Non-Goals:**

- 不支援複選題、問答題或超過 A/B/C/D 四個選項。
- 不處理跨課堂歷史分析報表與長期資料倉儲。
- 不變更現有身份認證與加入課堂流程。

## Decisions

### 1) 快問快答採用獨立事件族群，但沿用既有 session realtime channel

- 決策：在既有 `session-realtime-channel` 上新增快問快答事件型別（例如 `quick_qa_published`、`quick_qa_answered`、`quick_qa_closed`、`quick_qa_result_updated`）。
- 理由：可重用現有連線、廣播、重連機制，降低引入新通道風險。
- 替代方案：另開專用 websocket channel。
- 不採用原因：增加連線管理複雜度，且對目前單一課堂互動場景收益有限。

### 2) 題目與選項欄位允許空字串，但選項鍵固定為 A/B/C/D

- 決策：資料模型固定四個選項鍵，文字可為空。
- 理由：符合教師口頭敘述需求，同時保留學生端一致 UI 及簡化驗證邏輯。
- 替代方案：允許動態選項數量。
- 不採用原因：會增加前後端驗證、排序與統計複雜度，不符合目前需求。

### 3) 作答結果採「彙總預設、明細按需」的揭露策略

- 決策：教師端結果頁預設只顯示各選項人次/比例，學生名單需透過按鈕或分頁切換顯示。
- 理由：符合「不一定揭露個別答題狀態」的教學情境。
- 替代方案：初始畫面同時顯示統計與明細。
- 不採用原因：違反隱私揭露預期，也會增加資訊噪音。

### 4) 正確答案為可選欄位，學生回饋文案由結束事件帶入

- 決策：結束作答時可帶 `correctOption` 或 `null`；學生端僅在非空時顯示答對/答錯回饋。
- 理由：精準符合「可選擇正確答案或不選擇」需求，避免不必要評分提示。
- 替代方案：強制必填正確答案。
- 不採用原因：不符合教師僅做意見調查或暖身提問場景。

### 5) 學生作答採「截止前可改答，最後一次有效」

- 決策：同一學生在題目結束前可重複送出不同選項，伺服器以最後一次有效提交覆寫先前答案。
- 理由：符合課堂互動常見的再思考改答情境，且可減少誤觸後無法修正的挫折。
- 替代方案：僅首次作答有效。
- 不採用原因：會限制教學互動彈性，與需求「可隨時更換選擇」不符。

## Risks / Trade-offs

- [風險] 學生晚到或重連時看不到當前題目狀態 → [緩解] 進入課堂與重連成功後由伺服器主動回補目前快問快答狀態快照。
- [風險] 同一學生短時間頻繁改答造成統計跳動或覆寫競爭 → [緩解] 以學生識別鍵做最後一次作答覆寫，並以事件時間戳或序號保證最終一致。
- [取捨] 不顯示明細可降低揭露風險，但教師查看個別資料多一步操作 → [緩解] 提供清楚可見的「查看明細名單」切換入口。
- [風險] 正確答案設定時機不一致（先結束後補設）造成學生端訊息閃動 → [緩解] 以單一「結束事件」原子化帶入最終正解設定。

## Migration Plan

1. 先新增前後端型別與事件常數，確保通道可承載快問快答 payload。
2. 實作教師端快問快答頁面與結果檢視切換，不影響既有白板路由。
3. 實作學生端作答區塊與結束後回饋 UI。
4. 串接 Rust 廣播與狀態快照回補，完成整體同步。
5. 以 feature flag 或課堂模式開關逐步啟用，必要時可回退為不顯示快問快答入口。

回滾策略：若上線後發生同步異常，可先關閉教師端入口與事件發送，保留既有白板與課堂連線能力。
