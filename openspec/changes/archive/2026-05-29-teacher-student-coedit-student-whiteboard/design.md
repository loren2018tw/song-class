## Context

目前教師端可在學生白板分頁查看全班縮圖，學生端可將學生白板增量事件回傳教師端，教師端再更新縮圖；但教師無法針對單一學生進入可操作的共編畫面。現況缺口在於：

- 缺乏「單一學生白板」的聚焦編輯介面。
- 同一張學生白板只支援學生輸入，尚未支援教師輸入並回推學生端。
- 教師端縮圖、教師共編視圖與學生端白板間尚未建立同源狀態同步保證。

本次變更需在既有 WebRTC DataChannel 架構下，擴充指定學生白板的雙向即時同步，並維持既有全班縮圖能力。

## Goals / Non-Goals

**Goals:**

- 教師端點擊學生縮圖後，開啟大型對話框（約 80% 寬）顯示目標學生白板，並提供工具列與暱稱標題。
- 教師可在該對話框直接繪製，事件即時傳送到對應學生端並立即反映。
- 學生原有繪製回傳機制維持可用，且能與教師輸入共同作用於同一白板。
- 保證三個視圖一致：學生端白板、教師端縮圖、教師端共編對話框。
- 對話框可隨時關閉且不破壞其他學生的白板同步。

**Non-Goals:**

- 不實作多教師同時共編同一學生白板。
- 不新增長期歷史版本或回放時間軸。
- 不變更首頁模式或教師白板（teacher-board）既有同步語意。
- 不標註教師與學生筆跡來源差異（例如顏色標記、來源標籤或圖例）。

## Decisions

### 決策 1：以「指定學生通道 + 指定白板序列」擴充事件協定

- 決策：在既有 `WhiteboardSyncMessage` 中新增教師對學生白板的事件批次訊息，帶入 `targetStudentId`（或由點對點 channel 隱含目標）與 `student-board` 序列欄位，沿用批次與序號校驗模型。
- 理由：可重用既有增量重播與序列斷裂處理邏輯，降低實作風險。
- 替代方案：
  - 直接傳整份快照：簡單但頻寬成本高、延遲大。
  - 為共編開新 DataChannel：隔離度高但會增加協商與生命週期管理複雜度。

### 決策 2：教師端對話框採用「來源即真相」的本地快照鏡像

- 決策：教師端共編對話框顯示內容與縮圖共用同一份 `studentBoardSnapshots[targetId]` 狀態；教師落筆時先更新本地該快照，再送出增量訊息。
- 理由：可確保縮圖與對話框在教師端零延遲一致，並降低雙狀態漂移。
- 替代方案：對話框維護獨立 snapshot 再回寫縮圖，會提高狀態同步錯誤機率。

### 決策 3：學生端採「單白板事件合併重播」

- 決策：學生端在 `student-board` 同時接受本地繪製與教師遠端繪製事件，統一進入同一重播管線並維護單一 sequence。
- 理由：同一白板只保留一條序列，最容易保證一致性與補償邏輯。
- 替代方案：本地序列與遠端序列分開再合併，容易出現衝突與順序不穩定。

### 決策 4：關閉對話框僅終止教師 UI 互動，不中斷學生連線

- 決策：關閉共編對話框時只清除 `activeCoeditStudentId` 與相關 UI state，不關閉該學生的 peer/channel。
- 理由：避免重開時重新協商，保持縮圖持續同步。
- 替代方案：關閉即中斷該學生通道，會導致額外延遲與不必要重連。

### 決策 5：共編對話框支援 Esc 鍵快速關閉

- 決策：教師端共編對話框必須支援鍵盤 `Esc` 快速關閉，並與點擊關閉按鈕採相同行為。
- 理由：提升課堂切換節奏與操作效率，減少滑鼠移動成本。
- 替代方案：僅保留滑鼠關閉，實作較簡單但操作效率較差。

### 決策 6：不提供教師筆跡來源標註

- 決策：共編白板採單一視覺語意呈現，不額外標示教師筆跡來源。
- 理由：降低視覺複雜度，維持學生白板版面清晰。
- 替代方案：加入來源標記或獨立顏色規則，辨識度較高但會增加 UI 與教學干擾。

## Risks / Trade-offs

- [風險] 教師與學生同時高頻輸入造成事件交錯，可能引發短暫筆跡順序感知差異 → [緩解] 以單序列重播與序列缺口補快照機制統一收斂。
- [風險] 教師端同時維護多位學生縮圖與單人共編，渲染負載上升 → [緩解] 共編視圖僅針對單一學生啟用工具列互動，其他維持無工具列縮圖。
- [風險] 目標學生離線時對話框仍開啟造成操作歧義 → [緩解] 學生離線事件觸發對話框自動關閉與提示訊息。
- [取捨] 優先沿用現有訊息模型，短期內訊息型別會變多 → [緩解] 以明確型別命名與守衛函式維持可讀性。

## Migration Plan

1. 擴充 `src/types/whiteboard.ts` 訊息型別與 type guard，保持舊訊息可相容解析。
2. 先實作教師端對話框 UI 與目標學生狀態管理，確認不影響既有縮圖展示。
3. 實作教師→學生的指定白板增量同步，再串接學生端重播。
4. 以整合測試驗證三視圖一致性、離線回收與關閉對話框行為。
5. 若上線後發現同步異常，可切換 feature flag（或暫時隱藏共編入口）回到純縮圖監看模式。
