## Context

現有架構已具備教師與學生間的 WebRTC 連線與 DataChannel 事件傳遞能力，但目前缺乏教師螢幕媒體串流流程，以及「白板模式 / 快速問答 / 廣播教師畫面」三模組的互斥控制。若讓多模組同時啟用，容易造成學生端畫面競爭、事件覆蓋與教師流程混亂。

主要限制與關係人：

- 教師需要單一按鈕啟停廣播並由系統觸發螢幕擷取授權流程。
- 學生端需在廣播時穩定接收教師串流，並於停止後回到正確模組畫面。
- 白板模式、快速問答與廣播模組必須互斥，不得互相干擾。
- 即時通道需維持向後相容，避免影響既有協作白板與快問快答流程。
- 本階段部署前提為教師與學生在同一區域網路（LAN），先維持 P2P 架構。

## Goals / Non-Goals

**Goals:**

- 實作教師端螢幕擷取與 WebRTC 媒體串流下發給所有學生。
- 建立模組互斥機制，確保同一時間僅有一種教學模組啟用。
- 教師端提供清楚的廣播控制與廣播中提示。
- 學生端可自動切換到串流接收畫面並穩定播放教師串流。
- 新加入或重連學生可在加入同步後拿到目前模組狀態並完成串流接收。

**Non-Goals:**

- 本次不實作伺服器中繼（SFU/MCU）架構，維持教師對每位學生點對點下發。
- 本次不新增錄影、回放、或學生端反控功能。
- 本次不調整白板筆跡資料結構與快問快答統計演算法。
- 本次不處理跨網際網路大規模佈署最佳化（例如 TURN 成本與 SFU 擴充）。

## Decisions

1. 以 session 層級 `activeModule` 作為唯一來源真相，值包含 `home | whiteboard | quick_qa | teacher_screen_broadcast`。

- 原因：可用統一狀態保證模組互斥，避免白板與快問快答、廣播同時生效。
- 取代方案：各模組各自維護布林旗標。未採用，因為容易出現多旗標同時為真。

2. 廣播啟動時教師端呼叫瀏覽器螢幕擷取 API，將取得的 video track 掛入既有師生 RTCPeerConnection，透過 renegotiation 下發媒體。

- 原因：沿用現有 WebRTC 連線可降低基礎設施變更與重建連線成本。
- 取代方案：為廣播另建一組獨立 peer connection。未採用，因會增加連線數與狀態同步成本。

3. 控制事件與媒體事件分離：DataChannel 傳送模組切換與廣播控制，MediaStream 只承載影像資料。

- 原因：語意清楚、除錯容易，並可保留控制面向向後相容。
- 取代方案：僅靠媒體軌是否存在判斷狀態。未採用，因狀態推導不穩且晚加入難以初始化。

4. 加入同步（bootstrap）必須攜帶 `activeModule` 與廣播 metadata，若當前為廣播模式則新加入學生需觸發媒體重協商。

- 原因：確保晚加入學生可直接進入正確畫面並接收串流。
- 取代方案：學生加入後輪詢教師狀態。未採用，因延遲高且流程複雜。

5. 為控制 P2P 廣播上行壓力，教師螢幕串流預設採固定低延遲參數（建議 720p / 10-15fps），並保留依學生數量降規的設定鉤子。

- 原因：在同區網下可先以 P2P 交付，並透過參數限制降低教師端上行與編碼負擔。
- 取代方案：立即導入 SFU。未採用，因目前場景限定同區網且要先維持既有架構。

## Risks / Trade-offs

- [風險] 教師未授權螢幕擷取或中途關閉分享。 → Mitigation：捕捉 permission 與 track ended 事件，立即停止廣播並回退到 home。
- [風險] 多學生同時接收串流造成教師端上行頻寬壓力。 → Mitigation：限制解析度與幀率，並監控連線品質；必要時後續導入 SFU。
- [風險] 同區網品質不佳時仍可能出現卡頓。 → Mitigation：提供一鍵降畫質設定（例如 540p / 10fps）與教師端提示。
- [風險] 模組切換時殘留舊事件造成跨模組干擾。 → Mitigation：所有模組入口先經 `activeModule` 切換流程，切換時清理不相容事件處理器。
- [風險] 舊版客戶端不認得新事件。 → Mitigation：未知事件忽略，加入同步缺值時回退為 home。

## Migration Plan

1. 擴充 session 型別與事件協定，新增 `activeModule` 與廣播控制/協商事件。
2. 實作教師端啟停廣播流程：getDisplayMedia、track 掛載、renegotiation、停止清理。
3. 實作學生端串流接收畫面與模組切換邏輯。
4. 將白板與快問快答入口接入模組互斥守門，確保切換時不互相干擾。
5. 針對新加入學生補上同步與重協商流程。
6. 回滾策略：若異常可關閉廣播入口並將 `activeModule` 強制回 home，既有白板與快問快答維持可用。

## Open Questions

- 廣播中是否需要在教師端顯示各學生串流接收成功數量？
- 是否需要在低頻寬時自動降畫質，或先以固定畫質實作？
- 若未來超出同區網場景，是否啟動 SFU 遷移計畫作為下一階段變更？
