## Context

目前教師端雖已有多種課堂管理功能，但缺少「聯絡簿顯示」與「作業繳交控管」的一體化任務管理。現況下教師需分散處理任務公告與繳交追蹤，無法以同一筆資料同時支援兩種教學情境。此變更需在不破壞既有課堂流程下，導入可共用資料模型、雙標籤 UI 與學生繳交狀態追蹤。

主要約束如下：

- 前端採 Vue 3 + Vuetify，教師端 Web 視圖需延續既有版型與互動習慣。
- 後端為 Rust + Tauri 架構，資料讀寫需與既有班級/學生會話資訊整合。
- 開發期資料庫版本維持 v0，結構調整以 DDL 與命令列執行，不在應用程式內寫自動 migration。

## Goals / Non-Goals

**Goals:**

- 建立單一任務資料模型，讓同一筆任務可同時驅動聯絡簿顯示與作業繳交管理。
- 在「聯絡簿管理」頁提供「聯絡簿」與「作業繳交管理」雙標籤，且兩分頁皆可新增任務。
- 支援任務編輯、刪除、顯示條件與勾選規則（不可同時取消「顯示在聯絡簿」與「是否需控管」）。
- 對需控管任務提供學生層級繳交狀態，並可在 UI 區分已繳交與未繳交。
- 在作業繳交管理提供完成狀態篩選（全部、未完成、已完成）與「不限日期未完成」切換以提升查找效率。
- 聯絡簿與作業繳交管理皆提供日期選擇器，預設僅顯示所選日期任務。
- 所有任務與繳交管理資料只作用於目前選擇班級。

**Non-Goals:**

- 不在本次變更導入完整歷史版本化 migration 機制。
- 不處理跨班級批次任務建立或跨日期批次複製。
- 不新增學生端上傳檔案流程（僅定義教師端任務/狀態管理）。

## Decisions

1. 採用單一「任務」主表 +「任務繳交狀態」從表

- 決策：任務主表保存共通欄位（id、class_id、date、title、show_in_contact_book、requires_tracking、is_completed），從表保存 task_id + student_id + submitted 狀態。
- 原因：符合「聯絡簿顯示項目也是作業」需求，避免維護兩套重複資料。
- 替代方案：分成聯絡簿表與作業表再做同步。未採用原因：同步複雜、容易資料不一致。

2. 以旗標驅動雙標籤顯示

- 決策：聯絡簿標籤只顯示 show_in_contact_book=true 的任務；作業繳交管理標籤只顯示 requires_tracking=true 的任務。
- 原因：直接對應使用者心智模型，降低額外映射邏輯。
- 替代方案：依任務類型欄位分類。未採用原因：無法同時滿足「可同時勾選」的需求。

3. 在任務建立與編輯時做雙旗標驗證

- 決策：前後端都必須驗證 show_in_contact_book 或 requires_tracking 至少一者為 true。
- 原因：避免資料層出現無法在任何分頁顯示的孤兒任務。
- 替代方案：僅前端限制。未採用原因：無法防止 API 直接寫入無效資料。

4. 完成狀態採任務層聚合欄位

- 決策：保留任務層 is_completed 作為穩定完成快照，並由學生繳交狀態自動推導同步。
- 原因：班級成員可能刪增，若只即時計算目前名單會讓已完成任務在名單變動後回退為未完成，造成教師判讀混亂；保留欄位可維持任務完成狀態穩定且可追蹤。
- 替代方案：每次依子表計算完成。未採用原因：查詢成本高且前端交互延遲較大。

5. 日期篩選與不限日期未完成切換

- 決策：聯絡簿與作業繳交管理都提供日期選擇器，預設只顯示所選日期任務；另在作業繳交管理提供「顯示不限日期所有未完成」切換。
- 原因：教師日常查看通常以當日任務為主，但追未完成任務需要跨日期視角。
- 替代方案：僅提供日期選擇，不提供跨日期未完成。未採用原因：追蹤補交作業效率不足。

6. 僅對目前班級作用

- 決策：查詢、建立、更新、刪除與繳交狀態統計都嚴格限定於目前選擇班級，不提供跨班排序或聚合。
- 原因：符合實際使用情境與需求邊界，避免跨班資料混淆。
- 替代方案：提供跨班清單與排序。未採用原因：超出本次變更範圍。

## Risks / Trade-offs

- [任務完成與學生繳交狀態可能短暫不一致] → 透過同一 API 交易更新或後端一致性檢查，確保批次更新後狀態同步。
- [雙標籤共用新增按鈕導致重複互動定義] → 抽離共用任務表單元件，兩標籤只保留入口與預設值。
- [班級人數較多時繳交清單渲染壓力上升] → 使用分區塊渲染與必要欄位載入，避免一次載入過多細節。
- [v0 開發期 schema 易變動] → 明確將 DDL 變更集中於 sql/v0.sql，並在任務中加入驗證步驟。
- [不限日期未完成查詢資料量增加] → 預設關閉該切換，且查詢僅返回目前班級未完成任務。

## Migration Plan

1. 以命令列更新/套用 v0 DDL，新增任務與任務繳交狀態資料表（或補齊欄位）。
2. 後端先實作查詢與寫入 API，加上雙旗標防呆驗證，並實作由繳交狀態自動推導 is_completed。
3. 前端接入新 API，完成雙標籤畫面、日期選擇、任務 CRUD、完成狀態篩選與繳交標記。
4. 使用既有測試資料驗證：
   - 只顯示在聯絡簿
   - 只需控管
   - 同時顯示+控管
   - 不可雙取消
   - 日期篩選與不限日期未完成切換
   - 班級切換後僅顯示目前班級資料
5. 若需回滾：先停用新 UI 入口，再回退 API 與 DDL（刪除新增欄位/表前需先備份資料）。
