## Context

目前系統已具備教師端、學生端與主控台三種介面，並透過內建 Web Server 與 WebSocket 管理即時課堂互動。學生加入流程現況為輸入自由暱稱，不具備班級名單約束；主控台亦缺少班級資料來源與編修入口。此次變更需導入 SQLite 作為班級/學生資料持久層，並以「班級名單選擇」取代自由輸入，讓教師在真實班級情境下可管理固定座號名單與連線占用狀態。

限制條件：

- 專案目前處於資料庫結構版本 0，僅允許初始化，不導入 migration 流程。
- DDL 需以檔案形式提供，初始化時僅在資料庫不存在或無班級資料時建立預設資料。
- 前端需沿用 Vue 3 + Vuetify，並維持教師端/學生端既有即時連線模型。

## Goals / Non-Goals

**Goals:**

- 建立 SQLite 班級資料模型與 v0 DDL，至少涵蓋班級與學生名單。
- 啟動時確保存在預設班級與 30 位學生（座號 `01`~`30`，暱稱預設空字串）。
- 主控台新增班級管理區塊，提供班級選擇與編輯班級成員入口，且可修改座號與暱稱。
- 學生端改為顯示「目前班級學生按鈕」加入，並鎖定已連線學生按鈕。
- 教師端頁面上方顯示目前班級名稱。
- 網頁中所有學生名稱顯示統一為「座號暱稱」。
- 切換班級時強制登出該班在線學生，避免身份殘留到新班級。
- 首個由系統建立的班級與一般班級等價，名稱可被編輯。

**Non-Goals:**

- 不處理多班級同步授課或跨班級同時廣播。
- 不在本次導入資料庫 schema 版本升級機制或 migration 歷史表。
- 不實作複雜權限系統（例如班級編輯者角色分級）。

## Decisions

### 1. 使用 SQLite 檔案資料庫並新增 v0 DDL 檔

- 決策：在 Rust 後端啟動流程加入「資料庫存在檢查 + 套用 v0 DDL」初始化路徑，DDL 以獨立檔案維護。
- 理由：符合專案開發期規範，避免把 schema 建立邏輯散落在程式碼中，且便於命令列重建資料。
- 取代方案：
  - 以程式碼動態 `CREATE TABLE IF NOT EXISTS`：被拒絕，違反開發規範。
  - 改用遠端 DB（PostgreSQL）：被拒絕，部署與開發成本過高。

### 2. 以 `classrooms` + `students` 二表模型承載班級名單

- 決策：
  - `classrooms`：`id`, `name`, `created_at`, `updated_at`
  - `students`：`id`, `classroom_id`, `seat_no_text`, `nickname`, `created_at`, `updated_at`
  - `seat_no_text` 對同一班級唯一，固定使用兩位文字座號。
- 理由：可直接映射 UI 的班級/座號概念，並支援後續班級切換與成員編輯（含座號與暱稱）。
- 取代方案：
  - 單表存 JSON 名單：查詢與唯一性驗證困難。
  - 以整數存座號：無法保留 `01` 這類顯示格式，仍需額外轉換。

### 3. 網頁顯示名稱統一使用「座號暱稱（無分隔）」

- 決策：所有 Web UI 顯示學生身分文字時，統一由後端或共用 formatter 輸出 `seat_no_text + nickname`（無空白、無符號分隔），例如 `01王小明`；暱稱為空時僅顯示座號。
- 理由：避免不同頁面混用名稱格式造成教師辨識困難，並確保與班級名單一致。
- 取代方案：
  - 各頁面各自拼接顯示字串：容易出現格式不一致與遺漏。
  - 僅顯示暱稱：暱稱可空白且可能重複，不利辨識。

### 4. 已連線學生按鈕鎖定使用「即時連線占用 + roster key」

- 決策：學生端按鈕可用性由「目前 class + seat_no_text」組合鍵決定；當同組合鍵已在線，按鈕 disabled。
- 理由：避免單純用暱稱比對造成重名衝突，也能精準對應班級名單項目。
- 取代方案：
  - 只鎖定暱稱：暱稱可空白，無法當唯一識別。
  - 客戶端本地鎖定：斷線/重連時一致性不足。

### 5. 班級切換採「強制登出在線學生」策略

- 決策：切換目前班級時，後端先廣播並執行舊班級在線學生強制登出，再切換班級上下文並推送新班級狀態。
- 理由：可避免舊班級會話殘留、按鈕占用錯誤與跨班錯寫資料。
- 取代方案：
  - 切換班級後保留舊連線：容易造成身分與班級不一致。
  - 僅前端隱藏舊班學生：後端狀態仍殘留，不可接受。

### 6. 主控台與教師端共用「目前班級狀態來源」

- 決策：由後端維護目前班級 ID/名稱，主控台切換班級後透過現有狀態通道推送給教師端與學生端。
- 理由：避免前端各自管理造成狀態分裂，且可沿用既有即時事件機制。
- 取代方案：
  - 各前端輪詢 REST：延遲較高且增加 API 成本。

## Risks / Trade-offs

- [風險] 班級切換觸發強制登出可能造成學生體驗中斷。→ Mitigation：提供明確提示訊息，並在新班級狀態同步後立即顯示可重新加入名單。
- [風險] 預設班級初始化重複執行造成重複學生資料。→ Mitigation：以唯一鍵（`classroom_id`, `seat_no_text`）保護，初始化使用交易與 upsert/insert-ignore。
- [取捨] 座號採文字儲存可保留顯示格式，但排序需明確使用字串規則（`01`~`30`）。
- [取捨] 使用單一目前班級狀態簡化系統，但限制同時多班操作場景。

## Migration Plan

1. 新增 v0 DDL 檔案（包含 `classrooms`、`students` 及必要索引）。
2. 在後端啟動流程加入 DB 初始化：
   - 若 DB 檔不存在，建立檔案並套用 DDL。

- 若 `classrooms` 無資料，建立系統協助建立的首個班級與 30 位預設學生。

3. 新增班級查詢/切換 API、班級成員編輯（座號與暱稱）能力與事件廣播。
4. 實作班級切換強制登出流程與狀態重算。
5. 更新主控台 UI（班級顯示、班級選擇、成員編輯入口）。
6. 更新教師端與其他 Web 頁面學生名稱顯示格式為「座號暱稱」。
7. 更新學生端加入流程為名單按鈕，並串接 disabled 狀態。
8. 驗證：重啟後資料持久、班級切換強制登出生效、名稱顯示格式一致、已在線學生不可重複點選。

Rollback 策略：

- 前端可回退至暱稱輸入流程版本。
- DB 可保留檔案但不啟用班級選擇入口（功能旗標或回退 commit）。
- 若 DDL 發生問題，刪除開發期 DB 後以修正版 DDL 重建。
