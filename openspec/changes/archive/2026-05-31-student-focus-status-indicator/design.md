## Context

目前教師端左側連線清單僅呈現學生是否連線，缺少「是否仍在學生端分頁」的可視化訊號。需求需同時涵蓋學生端分頁狀態偵測、即時事件傳輸、教師端狀態儲存與 UI 呈現，屬於跨越前端元件、通訊事件契約與型別定義的變更。

限制條件：

- 必須沿用現有 Vue 3 + Vuetify 與既有即時通道，不新增外部服務。
- 分頁焦點狀態是提示性訊號，不能阻斷學生繼續使用功能。
- 狀態需能在學生切換分頁後快速反映，且在斷線/重連時保持可預期。

## Goals / Non-Goals

**Goals:**

- 在學生端偵測分頁是否可見並即時送出狀態事件。
- 在教師端維護每位學生最新專心狀態，並以暱稱標籤底色區分。
- 定義一致的狀態生命週期（加入、切離、切回、離線、重連）。
- 讓狀態顯示與現有學生清單/連線狀態共存，不破壞既有互動流程。

**Non-Goals:**

- 不加入教師端主動警示音效、彈窗或懲罰機制。
- 不追蹤學生切去哪個網站或頁面內容。
- 不提供歷史專注度報表與長期統計儲存。

## Decisions

1. 以瀏覽器 `visibilitychange` 為主要訊號來源，`visible` 對應「專心中」、`hidden` 對應「已離開分頁」。

- 理由：跨平台支援成熟，語意直接對應「是否停留在本分頁」。
- 替代方案：`window blur/focus` 容易受視窗切換與作業系統行為影響，誤判率較高，因此不採用為主訊號。

2. 狀態事件沿用既有即時通道，新增學生焦點狀態事件型別，不另開新連線。

- 理由：可重用既有連線管理與重連機制，降低實作與維運成本。
- 替代方案：獨立心跳 API 會增加同步複雜度與狀態分裂風險。

3. 教師端以「最後事件覆寫」維護每位學生當前狀態，預設值為加入時「專心中」。

- 理由：該需求是即時顯示目前狀態，不需保留完整事件歷史。
- 替代方案：事件序列重播可提供追溯能力，但此變更不需要，且會提高儲存與處理成本。

4. UI 以 Vuetify `v-chip`（或等效標籤元件）底色呈現雙態顏色，並維持文字可讀性對比。

- 理由：符合既有元件系統且容易在桌機/行動瀏覽器維持一致樣式。
- 替代方案：僅用圖示或文字狀態辨識度較低，不利教師快速掃描。

## Risks / Trade-offs

- [風險] 部分行動瀏覽器在背景化時事件延遲或節流，導致狀態更新不即時。 → [緩解] 教師端顯示「目前狀態為近即時訊號」語意，並在重連時重送最新狀態。
- [風險] 使用者短暫切換 app 可能造成頻繁顏色閃動。 → [緩解] 前端可加入極短去抖（例如數百毫秒）避免抖動。
- [取捨] 只保留最新狀態可簡化系統，但無法直接做課後分析。 → [緩解] 後續若有分析需求，再另提變更加入歷史紀錄能力。

## Migration Plan

- 屬於向後相容的增量變更，先部署事件型別與教師端容錯（未知事件忽略）。
- 逐步啟用學生端事件發送與教師端 UI 顯示。
- 若需回滾，可停用學生端事件發送與教師端標籤顏色映射，不影響核心連線流程。

## Open Questions

- 是否需要在教師端同時顯示「最後狀態更新時間」以協助判讀延遲？
- 當學生網路中斷但頁面仍前景時，教師端是否一律以離線優先顯示而非專心狀態？
