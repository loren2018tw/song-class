## Context

目前教師端已有即時互動模組（白板、快問快答、聯絡簿）與學生清單資料來源，具備可重用的班級狀態與連線基礎，但尚未提供「積點」這類長時段、可重設、可群組套用的課堂管理能力。本次設計需同時滿足兩種操作模式：

- 個人積點：針對個別學生或全班進行加減與重設。
- 小組模式：以拖拉方式分組，並對群組全員加減。

約束條件：

- 前端需維持 Vue 3 + Vuetify 元件化實作，並與既有教師端操作風格一致。
- 分組與積點資料以學生資料表欄位為主，不額外建立分組設定檔持久化。
- 若有資料表結構調整，必須由命令列操作並記錄於 `src-tauri/sql/migrations/next_release.sql`，保留升版路徑。

## Goals / Non-Goals

**Goals:**

- 在教師端提供「學生積點模組」並以 tab 區分個人積點與小組模式。
- 建立可持久化的學生積點資料模型，支援單人、全班、整組三種更新粒度。
- 建立小組分配資料模型，支援拖拉調整且即時寫回學生組別欄位。
- 在小組模式提供可隨時調整的「分幾組」輸入框，並支援縮減組別時自動移入未分組。
- 規範資料結構異動流程：先命令列調整，再寫入 next_release.sql。

**Non-Goals:**

- 不在本次設計中定義複雜評分規則（如加權、積分兌換、歷史排行榜）。
- 不處理跨課堂/跨班級的積點匯總分析報表。
- 不實作學生端自主管理分組或修改分數權限。

## Decisions

1. 模組整合策略：在教師主頁維持單一入口，新增「學生積點模組」作為一級功能。

- 原因：教師操作集中、學習成本低，且可沿用現有班級上下文與連線狀態。
- 替代方案：獨立頁面路由。缺點是切換成本高，且需重建共用狀態。

2. UI 結構策略：模組內採雙分頁（個人積點、小組模式）。

- 原因：兩種任務模型差異大（卡片操作 vs 拖拉分組），分頁能降低互相干擾。
- 替代方案：同頁折疊區塊。缺點是控制項密度過高、誤觸機率高。

3. 積點資料模型：以「班級 + 學生」為主鍵持久化目前分數，並提供原子加減操作。

- 原因：可支援高頻更新與簡潔查詢，且容易做全班/整組批次更新。
- 替代方案：僅前端記憶體狀態。缺點是重新整理或重連即遺失。

4. 小組資料模型：在學生資料表新增 group_no 欄位，畫面端每次載入時以 group_no 分群產生各組與未分組。

- 原因：符合需求「由學生資料直接分群」，資料來源單一且可避免雙寫不一致。
- 替代方案：獨立群組對照表。缺點是查詢與同步路徑更複雜。

5. 積點資料模型：在學生資料表新增 points 欄位，個人與小組加減分皆直接更新該欄位。

- 原因：個人模式與小組模式共用同一欄位，可確保顯示與計算一致。
- 替代方案：獨立積點歷程表再彙總即時分數。缺點是本期需求不需歷程，開發成本較高。

6. 小組模式編輯策略：分頁上方提供「分幾組」輸入框；組數減少時，超出範圍組別學生一律回收至未分組（group_no = 0）。

- 原因：符合教師即時調整教學分組的需求，並避免學生遺留在不可見組別。
- 替代方案：阻止縮減組數直到手動搬移。缺點是操作繁瑣。

7. 小組模式儲存策略：拖拉學生到不同組塊時立即寫回學生 group_no，不提供另存按鈕。

- 原因：教師操作心智模型為「所見即所得」，即時儲存可減少漏存風險。
- 替代方案：草稿模式加儲存按鈕。缺點是流程多一步且易忘記提交。

8. 資料庫變更流程：禁止直接在程式碼內隱式修改 schema，僅透過命令列與 migration 草稿落地。

- 原因：可追蹤、可審核、可於 release 前整批驗證升級路徑。
- 替代方案：啟動時自動建表。缺點是版本漂移風險高。

## Risks / Trade-offs

- [拖拉分組在學生數較多時操作負擔增加] → 以座號暱稱排序、群組區塊固定高度與快速回收機制降低操作成本。
- [整組加減若包含誤分組成員，會造成批次誤更新] → 每次群組操作前後提供成員可視化清單，並保留重設機制。
- [高頻加減分可能造成前後端狀態不同步] → 後端採單一真實來源，前端每次更新以回傳狀態覆蓋或增量同步。
- [調小組數時大量學生回收到未分組，教師可能誤操作] → 在變更組數時顯示受影響人數並提供二次確認（可依門檻觸發）。

## Migration Plan

1. 先確認是否需要新增/調整積點與分組資料表欄位。
2. 使用命令列完成資料結構調整草稿，不透過應用程式啟動時隱式建置。
3. 將異動內容寫入 `src-tauri/sql/migrations/next_release.sql`。
4. 以既有 release 基準資料庫驗證可升級至目標結構。
5. 發版前再將草稿複製為正式版本 migration（例如 `NNN_*.sql`）。

## Operation Notes

- 積分允許負值，不設定下限。
- 未分組固定以 `group_no = 0` 表示。
- 拖拉分組與群組/個人加減分均為即時儲存，不提供額外儲存按鈕。
- 學生積點模組的學生來源為「目前班級名單」，不依賴連線狀態。
- 調小組數時，組號超出新範圍的學生會即時移回未分組（`group_no = 0`）。
