## Context

目前聯絡簿內容僅在教師端管理畫面可見，缺少一個可由老師主動發布到各班 LINE 官方帳號的流程。現有系統已具備班級與聯絡簿任務資料，但班級資料尚未承載 LINE 設定，聯絡簿工具列也尚未提供同步入口。本次設計需滿足「不依賴 webhook、不需常駐對外服務」的限制，讓桌面應用在老師按下同步時一次性呼叫 LINE Messaging API 完成 rich menu 更新。

限制與關係人：

- 限制：桌面端可能離線、各班使用不同 OA、敏感憑證不得明文外洩。
- 主要使用者：教師（設定班級 OA、手動同步）、家長（點擊 OA rich menu 取得最後同步內容）。
- 既有相依：聯絡簿任務資料來源與班級選擇機制。

## Goals / Non-Goals

**Goals:**

- 讓每班可獨立配置 LINE OA 設定並安全保存。
- 在聯絡簿管理畫面提供一鍵同步，來源為按下當下畫面內容。
- 同步時自動建立或更新 rich menu，確保存在「聯絡簿」按鈕。
- 「聯絡簿」按鈕內容每次同步覆寫為最新文字。
- 不依賴 webhook 與外部常駐服務。

**Non-Goals:**

- 不建立家長身份綁定、家長資料表或授權分流。
- 不提供即時動態回覆（不在點擊時查資料庫）。
- 不在此次變更導入複雜訊息模板編輯器。

## Decisions

### 1) 班級資料擴充 LINE 設定欄位

- 決策：在班級層增加 `line_enabled`、`line_channel_access_token`、`line_channel_secret`、`line_rich_menu_id`。
- 理由：需求明確是每班自己的 OA，且 richMenuId 必須可持久化供後續更新。
- 替代方案：獨立 `line_accounts` 資料表。
- 不採用原因：目前範圍僅每班一組設定，直接擴充班級欄位可降低複雜度。

### 2) 同步內容採「最後一次覆寫」模型

- 決策：同步按鈕觸發時生成文字，直接覆寫 rich menu 聯絡簿按鈕 message action 內容。
- 理由：符合「不常駐、不對外」要求，家長點擊即可得到最後一次同步內容。
- 替代方案：點擊時 webhook 回呼即時組文。
- 不採用原因：違反本次限制且增加維運成本。

### 3) richMenuId 生命週期由系統自動管理

- 決策：同步前若無 richMenuId 先建立，若無效則重建並回填。
- 理由：避免老師手動查填，降低操作門檻。
- 替代方案：要求老師手動維護 richMenuId。
- 不採用原因：錯誤率高且可用性差。

### 4) 訊息長度超限採自動摘要降級

- 決策：超限時自動縮減為摘要內容，並回饋教師已套用摘要。
- 理由：保障同步流程成功率，避免 LINE API 因欄位限制失敗。
- 替代方案：超限直接失敗。
- 不採用原因：教師體驗不佳，需反覆手動修文。

### 5) UI 佈局採低入侵調整

- 決策：聯絡簿工具列在「新增任務」旁增加「同步到官方帳號」；班級編輯對話框新增 LINE 設定區塊。
- 理由：延續既有操作習慣，降低學習成本。
- 替代方案：獨立 LINE 設定頁。
- 不採用原因：多一層導覽，與每班日常維護情境不符。

## Risks / Trade-offs

- [LINE API 變更或限流] → 加入錯誤分類與可重試提示，保留原 richMenuId 作為回復基準。
- [敏感憑證外洩] → UI 遮罩顯示、輸出 log 去識別、避免在錯誤訊息回顯 token。
- [同步內容過長被截斷造成誤解] → 明確標示為摘要模式，並保留可調整摘要策略。
- [rich menu 結構被人工改壞] → 同步前檢查並重建聯絡簿按鈕區塊。

## Migration Plan

1. 新增 migration：擴充班級資料表 LINE 欄位，預設為停用與空值。
2. 更新班級 API 與前端型別，支援讀寫 LINE 設定。
3. 在聯絡簿管理模組新增同步按鈕與同步狀態回饋。
4. 實作 LINE rich menu 同步服務（建立、更新、重建、摘要降級）。
5. 進行整合測試：首次同步、既有更新、失效重建、超限摘要、錯誤提示。
6. 發版時提供設定說明與最小操作指引。

Rollback 策略：

- 若同步功能異常，可關閉 `line_enabled` 並回退 UI 入口；資料欄位保留不影響既有聯絡簿功能。

- 是否需要在同步成功後保存一份本地同步快照供稽核？
